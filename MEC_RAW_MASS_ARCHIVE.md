# MEC_RAW_MASS_ARCHIVE
## Raw Mass Archive for AICOS Meta-Efficiency Core
**Status:** non-canonical  
**Purpose:** preserve ideation, variants, speculative directions, and discarded/parked concepts without contaminating the AICOS canon.  
**Rule:** this file is **not** source of truth for validated knowledge. It is an archive of architectural raw mass.

---

## 0. Why this file exists

The goal of MEC is not to build a bigger memory store, but a more efficient system for:
- evidence intake
- candidate distillation
- contradiction handling
- transfer testing
- judgment under constraints
- minimal canon export

During design, many ideas emerged:
- memory layering
- invariant extraction
- counterexample-first review
- transfer testing
- reconsolidation
- curiosity/blind-spot artifacts
- homeostatic control
- variant lab / mutation in shadow mode

Some of these are strong enough for immediate implementation.
Others are useful only as future options or conceptual scaffolding.

This archive preserves all of that **outside the registry**.

---

## 1. Core design ambition

The target is a **meta-efficient learning and judgment system**.

Not:
- a chat memory feature
- a giant note dump
- a universal archive of everything
- an emotionally invasive user profile
- a direct registry rewrite

But:
- a runtime-sidecar subsystem
- evidence-driven and review-gated
- minimal context cost
- explicit separation of runtime and canon
- anti-drift by design

## 2. Preserved concept families

### 2.1 Layered learning
Preserved idea:
- raw events are not canon
- candidate artifacts are not canon
- judgment is not identical with invariant knowledge
- canon should remain very small

Working abstraction:
- Event Zone
- Candidate Zone
- Review Zone
- Judgment Zone
- Canon Zone

### 2.2 Invariants over memories
Preserved idea:
- the system should not only remember what happened
- it should extract minimal principles that explain repeated patterns

Examples:
- lifecycle-management patterns
- stale-state patterns
- transferability of operational rules across domains

### 2.3 Boundary objects
Preserved idea:
- every strong rule needs explicit limits
- no invariant without failure conditions
- local truths must not masquerade as global truths

### 2.4 Counterexamples as first-class objects
Preserved idea:
- knowledge should be challenged, not merely accumulated
- contradiction pressure matters
- failure boundaries matter more than rhetorical certainty

### 2.5 Reconsolidation
Preserved idea:
- knowledge stays revisable
- reactivation should allow split / weaken / stabilize / reject
- no frozen canon without revalidation path

### 2.6 Curiosity / blind-spot artifacts
Preserved idea:
- the system must retain structured open questions
- unresolved blind spots are valuable
- unanswered areas should remain visible, not disappear

### 2.7 Homeostatic control
Preserved idea:
- rare but crucial knowledge must not be pruned away
- dominant but shallow knowledge must be damped
- low-recency, low-value artifacts should be flagged

### 2.8 Variant lab
Preserved idea:
- alternative rule forms can be explored
- only in shadow mode
- never auto-merge into canon

## 4. Strongly accepted ideas

These are preserved because they remain strategically important.

### 4.1 Runtime and canon must remain separate
Accepted strongly:
- runtime artifacts stay outside registry paths
- proposal-only artifacts must never appear in canon
- no silent registry mutation

### 4.2 Export must be explicit
Accepted strongly:
- no auto-promotion by time
- no status trick that silently becomes canon
- export requires gates, review and proof references

### 4.3 Review is not the same as export
Accepted strongly:
- a reviewed candidate is still not canon
- runtime acceptance is not registry acceptance
- read-models may expose derived status without mutating stored candidate files

### 4.4 Counterexample pressure is mandatory
Accepted strongly:
- every strong candidate should face contradiction search
- transfer without challenge is weak
- canon without boundary is brittle

### 4.5 Context efficiency matters
Accepted strongly:
- raw logs should not be shoved into prompts
- active agent context should be short, curated, purpose-specific

## 5. Parked or deferred ideas

These ideas are not rejected forever, but are **not immediate MVP scope**.

### 5.1 Full judgment engine
Deferred because:
- current AICOS runtime memory path is still stabilizing
- judgment needs multiple stable inputs, not just candidate review

### 5.2 Full homeostatic control
Deferred because:
- useful later, but not required for first MEC sidecar MVP
- needs stable artifact traffic first

### 5.3 Full variant lab
Deferred because:
- high power, high drift risk
- should come only after review and export gates are solid

### 5.4 Cross-app person memory
Deferred because:
- privacy risk
- scope bloat
- not needed for first AICOS-sidecar build

### 5.5 Emotion-centric memory scoring
Deferred / softened because:
- strong metaphor, weak immediate necessity
- use neutral priority/salience scoring first

---

## 6. Rejected or constrained directions

### 6.1 Giant raw-mass registry card
Rejected because:
- contaminates canon
- confuses agents
- creates false authority
- hurts search and review quality

### 6.2 Direct registry rewrite
Rejected because:
- current repo has already established a clean runtime-outside-registry principle
- MEC should grow as sidecar, not as destructive rewrite

### 6.3 Auto-promotion by timer
Rejected because:
- violates human-gate principle
- creates silent drift
- confuses runtime acceptance with canon export

### 6.4 Monolithic “memory OS”
Rejected because:
- too broad
- mixes storage, learning, judgment and canon
- invites complexity without control

---

## 7. Current strategic decision

The preferred path is:

- keep AICOS registry as canon source of truth
- build MEC as a sidecar subsystem
- start with runtime-only artifacts
- add review and contradiction handling
- only later add explicit export to canon

This archive supports that direction, but does not override the locked architecture documents.
